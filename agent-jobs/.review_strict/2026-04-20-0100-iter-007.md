## Code Review

### `Sources/AgentJobsCore/Discovery/Providers/LaunchdUserProvider.swift` (95 lines)

**✅ Strengths**:
- `Runner` typealias is a `@Sendable async throws -> String` — proper Swift 6 concurrency-safe injection seam for tests, doesn't require a mock-protocol explosion.
- Real path goes through `Shell.run`, inheriting the iter-004 timeout/argv hardening.
- Pure `parse()` is `static` and accepts `String` — trivially unit-testable, and the 7 tests exercise exactly the failure modes (header line, Apple filter, idle vs running vs failed).
- `com.apple.*` filter is the right product call — shows user agents, not system noise.
- Errors are wrapped: `Shell.Failure → ProviderError.ioError(...)` with the launchctl exit code & truncated stderr.

**⚠️ Findings**:

| ID | Severity | Issue |
|----|----------|-------|
| M-006 | MEDIUM | `createdAt: Date()` — synthetic timestamp, violates [feedback_tui_history](MEMORY) which wants real registration time. **Acceptable** because launchctl doesn't expose load time, but should be `nil`-able or carry a "synthetic" flag so the UI knows not to display it as authoritative. |
| L-007 | LOW | `schedule: .onDemand` is the wrong default for launchd jobs — they may have `StartCalendarInterval` / `StartInterval` plist keys. To get real schedules we'd need `launchctl print gui/$UID/<label>` per-job (slow). For now `.onDemand` is honest about "we don't know"; document this. Violates [feedback_schedule_display](MEMORY) "show actual frequency". |
| L-008 | LOW | `kind: pid != nil ? .daemon : .scheduled` — a daemon vs scheduled distinction that requires the same plist parse to be accurate. Same defer-to-future trade-off. |
| L-009 | LOW | `command: ""` empty string — should be `nil` if Optional, or sentinel `"<unknown>"` to signal absence rather than "command is literally empty". |

### `ServiceRegistry.swift` diff

`providerCount: Int { providers.count }` exposed on the actor — clean read-only view. Used precisely once by the view model to distinguish "no providers configured (idle)" from "all providers returned empty (error)". ✅

`defaultRegistry()` now returns 2 providers — `AgentJobsJsonProvider` + `LaunchdUserProvider`. ✅ matches code-003 M2.

### `AgentJobsMacApp.swift` diff (M1 wire-up)

```swift
if discovered.isEmpty && providerCount > 0 {
    phase = .error("No providers responded")
} else {
    phase = .loaded
}
```

Logically correct. **But:** "all 2 providers returned empty" is not the same as "all providers errored" — a perfectly healthy machine with no agent-jobs.json AND no user launchd jobs would now show an error banner. Better:

- Track per-provider success vs failure inside `discoverAll()` and surface `.error` only when *every* provider threw, not when results merely happen to be empty.
- Or rename the message to "No services found — check that providers are configured" so it's not falsely accusing the providers of "not responding".

→ Filed as **M-007**.
