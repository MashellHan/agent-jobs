## Code Quality Observations

### 1. Name generation ‚Äî mostly correct

**detect.ts** ‚Äî The 14 BASH_PATTERNS produce reasonable names:
- `pm2 start server.js` ‚Üí `pm2 server.js` (good)
- `docker run -d --name myapp nginx` ‚Üí `myapp` (good ‚Äî prefers --name)
- `nohup node server.js &` ‚Üí `node server.js` (good)
- `launchctl load ~/Library/LaunchAgents/com.example.plist` ‚Üí `com.example` (good)

**Edge case (LOW):** `detect.ts:52` ‚Äî docker run without `--name` falls through to image name extraction. The regex `docker\s+run\s+(?:.*\s)(\S+)\s*$` is greedy and may capture flags-with-values as the "last arg" in edge cases like `docker run -d -e FOO=bar nginx:latest -extra`. The current test suite only covers `--name` and basic image detection, not adversarial docker CLI inputs. Risk is LOW because misidentification only affects the display name, not functionality.

**scanner.ts** ‚Äî `friendlyLiveName()` at line 67 handles the lsof-based name generation well. The priority chain (script file ‚Üí framework ‚Üí command+port) produces user-friendly names. The 12 unit tests cover all major branches.

### 2. Column alignment ‚Äî correct

### 3. Pattern matching edge cases

**detect.ts:78-93** ‚Äî The nohup pattern `\bnohup\s+(.+?)\s*&` correctly captures the inner command. However:

- **(LOW)** Line 84-86: The script arg finder looks for `.js|.mjs|.ts|.py|.rb|.go` extensions but not `.jsx`/`.tsx`. If someone runs `nohup node app.tsx &`, the label falls back to just the runtime name (`node`) instead of `node app.tsx`. Not a bug ‚Äî `.tsx` files are almost never run directly with `nohup` ‚Äî but an inconsistency with the `friendlyLiveName()` in scanner.ts which checks `.jsx|.tsx`.

- **(LOW)** Line 222: `isBackground` check uses `JSON.stringify(input)` to grep for `run_in_background`. This is technically correct but fragile ‚Äî if a tool_result contains the string "run_in_background" in its output, it would false-positive. The check should target `input.tool_input` or a specific field, not the entire serialized object.

### 4. Error handling at system boundaries

- **detect.ts:277-283** ‚Äî `readFileSync(0)` for stdin wrapped in try/catch. Correct for hook context but uses sync I/O which blocks the event loop. Acceptable because the hook is a short-lived process, not a server.

- **loader.ts:21-38** ‚Äî `loadRegisteredJobs()` handles both file-not-found and corrupt JSON silently (resolves `[]`). Correct for a dashboard that should degrade gracefully.

- **setup.ts:44-53** ‚Äî `loadSettings()` now handles corrupt JSON (try/catch added in review-013). Correct.

- **scanner.ts:93-134** ‚Äî `scanLiveProcesses()` swallows lsof errors and returns `[]`. The `err.stdout` extraction at line 95 is clever ‚Äî lsof returns exit code 1 even on partial success with some data. However, the type assertion `(err as { stdout?: string } | null)` is a pragmatic workaround for Node's `ExecFileException` type.
