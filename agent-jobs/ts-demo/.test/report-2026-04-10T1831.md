## 3. Code Quality Observations

### A. Name Generation Issues

**JSON residue bug (documented, not fixed):**
- `fixtures.ts:44-47` contains a fixture with JSON garbage in the name field: `'pm2 api.js"},"tool_result":"started process [api]\\nid: 0"'`
- The test at `job-table.test.tsx:129-139` explicitly documents this as a "bug baseline"
- The `truncate()` function at `utils.ts:89-92` faithfully truncates this garbage, producing unreadable output like `pm2 api.js"},"tool_result":"…`
- **Root cause:** This would happen if the detect hook's `HookInput` parsing received malformed JSON where tool_input and tool_result fields leaked into the name field. The current `detect.ts` pattern matchers extract clean names, so this bug appears to be a legacy issue from before structured pattern matching was implemented. The current code would not produce this name for a valid hook input.

**nohup pattern name quality (detect.ts:79-93):**
- When `nohup node server.js &` is detected, the label function correctly produces `"node server.js"` — this is good
- However, if the nohup inner command has no recognizable script file (e.g. `nohup ./my-binary &`), it falls back to just the runtime basename (e.g. `"my-binary"`), which loses the full path context

### B. Pattern Matching Edge Cases

**docker run regex (detect.ts:52):**
- The regex `docker\s+run\s+(?:.*\s)?-d\s` requires `-d` to be followed by a space. This would fail to match `docker run -d` at end-of-string (no trailing space). However, in practice Docker commands always have an image argument after `-d`, so this is unlikely to cause real issues.

**node/python script detection requiring server output (detect.ts:228-233):**
- For `node script.js` or `python script.py` commands, detection only triggers if the output matches `OUTPUT_PORT_RE` (listening/started/running/serving on a port) or the command ends with `&`. This is correct behavior to avoid false positives from build scripts.
- Edge case: A server that outputs something like "Ready on port 3000" without using the words "listening/started/running/serving" would not be detected.

**inferAgent false positive (scanner.ts:63):**
- `inferAgent` checks for substring "claw" to detect "openclaw". This means any command containing "claw" (e.g., `clawback-service`, a financial tool) would be misidentified as "openclaw". The test at `scanner.test.ts:129-131` confirms this behavior.

### C. Error Handling

**Good practices observed:**
- `detect.ts:130-135`: `scanLiveProcesses()` catches errors and resolves with `[]`
- `detect.ts:144-145`: `loadJobs()` catches parse errors and returns empty state
- `detect.ts:148-152`: `saveJobs()` uses atomic write (write to temp, then rename)
- `loader.ts:34`: `loadRegisteredJobs()` catches JSON parse errors
- `loader.ts:54`: `createWatcher()` catches fs.watch errors

**Potential issues:**
- `scanner.ts:95` — `(err as { stdout?: string } | null)?.stdout` is a fragile type assertion on Node's child_process error object. The actual error type from `execFile` is `ExecFileException` which extends `Error` and has a `stdout` property, so this works but the cast is unnecessarily loose.
- `setup.ts:48` — `loadSettings()` does `JSON.parse(readFileSync(...)) as Settings` with no validation. If settings.json contains valid JSON that doesn't match the Settings interface (e.g., `hooks.PostToolUse` is a string instead of an array), the code would throw at runtime when trying to iterate `.hooks.PostToolUse`. The review-012 notes this as M5.
- `cli/index.ts:71` — `require("../../package.json")` uses `createRequire` with `import.meta.url`, which works but is fragile if the built output directory structure changes.

### D. Logic Issues

**app.tsx cursor not clamped on tab switch (app.tsx:118-131):**
- When switching tabs with left/right arrows, `setCursor(0)` resets cursor to top. This is correct.
- However, if the filtered list becomes empty and the user presses up/down, the cursor guard `cursor < filtered.length - 1` evaluates to `cursor < -1` which is always false, preventing movement. This is correct but could be confusing — no crash though.

**Dedup by name only (detect.ts:178):**
- `registerJob` deduplicates by `name` only (`file.jobs.some((j) => j.name === label)`). Two different services with the same generated name (e.g., two different `flask-server` instances in different projects) would collide. The review-012 tracks this as M3.

**Race condition in detect.ts (detect.ts:133-151):**
- `loadJobs()` reads the file and `saveJobs()` writes it back. If two hook invocations run concurrently (which is possible since hooks are async), both could read the same state and one would overwrite the other's write. The atomic rename prevents partial writes but not lost updates. The review-012 tracks this as H2.

### E. Unused Parameters

- `header.tsx:9` — `currentTab` prop is declared in the `Props` interface and passed at `app.tsx:153` but never used in the `Header` component body.
