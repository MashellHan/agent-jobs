### Documented Known Issues (from review-013)

**1. JSON residue in job names (baseline bug)**
The `jsonResidueJob` fixture and test at `utils.test.ts:28-34` document that `truncate()` can produce unreadable output when a job name contains JSON garbage. The root cause is in how hook input was parsed historically -- the current `detect.ts` code correctly extracts the `command` field, so this would only occur from legacy data. No fix needed for current code.

**2. Registry write race (H2)**
`detect.ts` `loadJobs()`/`saveJobs()` uses read-modify-write without file locking. Two concurrent hook invocations could lose data. Mitigated by `async: true` in the hook config (serialization of async hooks), but not formally guaranteed. Not a test failure -- a theoretical race condition.

**3. `readFileSync(0)` for stdin (H4)**
`detect.ts` `main()` at line 279 reads stdin synchronously with `readFileSync(0)`. This works for pipe-based hook delivery but could block if called without piped input. The PostToolUse hook always pipes JSON to stdin, so this is safe in practice.
