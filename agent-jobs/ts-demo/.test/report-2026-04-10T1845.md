### 3.1 Name Generation

**Status:** Mostly clean. Two documented edge cases remain.

- **`detect.ts:79-92` (nohup label):** Correctly extracts script name (e.g., `node server.js`) from `nohup node server.js &`. Handles multiple runtimes and script extensions. No garbled names produced by current patterns.

- **Known issue (fixtures.ts:44-48):** The `jsonResidueJob` fixture documents a scenario where a job name contains JSON residue (`pm2 api.js"},"tool_result":"...`). This is a pre-existing data quality issue -- the detect hook correctly extracts the label from the regex match, but if a corrupt `jobs.json` file already exists with bad data, the display shows garbage. The `truncate` function then produces partial JSON in the output. This is documented as a "bug baseline" test in both `utils.test.ts:28-34` and `job-table.test.tsx:129-139`. The root cause would be in whatever external process corrupted the file, not in the current code paths.

### 3.5 Specific Bugs Found

**BUG-1 (MEDIUM): `setup.ts:48` -- no try/catch around `JSON.parse`**

```typescript
// setup.ts:48
return JSON.parse(readFileSync(SETTINGS_PATH, "utf-8")) as Settings;
```

If `~/.claude/settings.json` exists but contains corrupt JSON, `setup` and `teardown` will crash with an unhandled `SyntaxError`. Compare with `detect.ts:137-141` which wraps the same pattern in try/catch. This is inconsistent error handling at a system boundary.

**BUG-2 (LOW): `cli/index.ts:69-73` -- no error handling for `--version`**

```typescript
const { createRequire } = await import("module");
const require = createRequire(import.meta.url);
const { version } = require("../../package.json");
```

If the relative path to package.json is wrong (e.g., after restructuring), this crashes with no user-friendly error.

**BUG-3 (DOCUMENTED): `detect.ts:222` -- `run_in_background` detection via JSON.stringify**

```typescript
const isBackground = BACKGROUND_RE.test(cmd) || /run_in_background/.test(JSON.stringify(input));
```

This serializes the entire hook input to JSON and searches for the string `run_in_background`. This works but is fragile -- it would false-positive if the string `run_in_background` appears anywhere in the command output or file content. In practice this is unlikely but worth noting.

### 3.6 TypeScript Quality

- **No `any` usage** in source files. Type assertions use `as` with specific types.
- **Non-null assertions (`!`):** Used extensively (e.g., `m[1]!`, `fields[0]!`, `.pop()!`). These are safe given the preceding regex match guarantees captured groups exist, but could mask future pattern changes. Count: ~25 instances across detect.ts/scanner.ts.
- **`source` field typing (`detect.ts:238`):** The `source` parameter in `registerJob` is typed as `string` but the `Job.source` field is `JobSource = "registered" | "live" | "cron" | "launchd"`. The registered job uses `opts.source` which is passed as `"hook-bash"` or `"hook-file"` -- but this value is not stored on the `Job` object (the `JobsFile` type uses `Omit<Job, "source">`), so no type error occurs. The `source` param in `registerJob` is unused dead code.
