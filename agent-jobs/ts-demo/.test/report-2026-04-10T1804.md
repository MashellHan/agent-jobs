## Issues Fixed in This Report

### FIX 1: Snapshot tests failing due to time-dependent relative times (ROOT CAUSE FIX)

**Files changed:** `src/job-table.test.tsx`, `src/__snapshots__/job-table.test.tsx.snap`
**Prior state:** 2 snapshot tests failing every hour due to clock drift
**Root cause:** The snapshot tests render `formatRelativeTime()` output (e.g., "7h ago") but the test fixtures use hardcoded `created_at` timestamps. As wall-clock time advances, the relative time shifts (7h -> 8h -> 9h), causing snapshot mismatches. This is a **recurring structural problem** -- review-009 noted snapshots have been stale in v006, v007, v008, and v009, each time with a different proximate cause but the same underlying issue: no clock freezing.

**Fix applied:** Added `vi.useFakeTimers()` + `vi.setSystemTime(new Date("2026-04-10T17:00:00Z"))` in a top-level `beforeEach`/`afterEach` block in `job-table.test.tsx`. This freezes the clock at a time that produces stable relative times for all fixture timestamps. The snapshots were then regenerated with `--update`.

**Why this fix is permanent:** Unlike simply regenerating snapshots (which fixes them for ~1 hour), freezing the clock ensures the snapshot output is deterministic regardless of when tests run. This addresses the systemic pattern of stale snapshots noted across reviews v006-v009.

### Issues found

#### HIGH: scanner.ts has 16% statement coverage (lines 21-64, 93-170 uncovered)

**File:** `src/scanner.ts`
**Lines:** 21-64 (`parseLsofOutput`, `getFullCommand`, `inferAgent`), 93-170 (`scanLiveProcesses`, `scanClaudeScheduledTasks`)

Only `friendlyLiveName` is tested (12 tests). The core scanning functions (`parseLsofOutput`, `scanLiveProcesses`, `scanClaudeScheduledTasks`) and the `inferAgent` function have zero test coverage. `parseLsofOutput` has non-trivial parsing logic (splitting fields, PID dedup, port extraction from socket name strings). `inferAgent` has 5 branch paths.

**Recommendation:** Add unit tests for `parseLsofOutput` with mock lsof output strings (pure function, easy to test). Add tests for `inferAgent` covering all 5 branches (claude, cursor, copilot, openclaw, manual).

#### MEDIUM: detect.ts `main()` function untested (lines 278-301)

**File:** `src/cli/detect.ts:277-311`

The `main()` CLI entry point that reads stdin, echoes to stdout, parses JSON, and calls `detect()` is untested. This includes stdin read error handling, empty input handling, JSON parse error handling, and the stdout echo behavior (required by PostToolUse hook protocol).

#### MEDIUM: JSON residue in job names (documented bug)

**File:** `src/fixtures.ts:44-57`, `src/job-table.test.tsx:115-125`

The `jsonResidueJob` fixture documents a known issue where the detect hook stores raw JSON data in the job name field: `'pm2 api.js"},"tool_result":"started process [api]\\nid: 0"'`. The test at line 115 explicitly documents this as a "bug baseline". The truncation function handles it by cutting the string, but the underlying issue is that `detect.ts` may receive malformed input where `tool_input.command` contains JSON residue from the hook payload.

**Root cause analysis:** Looking at `detect.ts:218`, `cmd` is extracted from `input.tool_input?.command`. If the hook payload's JSON structure is malformed or the command field contains serialized nested JSON, the label functions will extract garbage. The regex patterns match correctly but the input is already contaminated.

**Recommendation:** Add input sanitization in `detect.ts` to strip JSON-like suffixes from the command string before pattern matching. A simple heuristic: if the command contains `"}` followed by `,"`, truncate at the `"}` boundary.

#### MEDIUM: No error handling in `loadSettings()` (`setup.ts:44-49`)

**File:** `src/cli/setup.ts:44-49`

`loadSettings()` uses `JSON.parse` without try/catch. If `~/.claude/settings.json` exists but contains malformed JSON, the function throws an unhandled exception that crashes the CLI. Compare with `detect.ts:loadJobs()` which correctly wraps the parse in try/catch.

#### LOW: `job-detail.tsx:32` branch not covered -- `job.pid` display

**File:** `src/components/job-detail.tsx:32`

The branch where `job.pid` is truthy (which adds a PID row to the detail panel) is not exercised in tests. The `normalJob` fixture has no `pid` field, and the `liveProcessJob` fixture (which has `pid: 12345`) is not used in the detail panel snapshot test.

#### LOW: `truncate` with `max=0` is potentially unsafe

**File:** `src/utils.ts:89-92`

`truncate("abc", 0)` returns `"…"` (a string of length 1, exceeding `max`). While unlikely to be called with max=0 in practice, the function contract is unclear at this boundary. The column widths all use `COL.x - 1` which is always >= 1.

#### LOW: `process.stdout.columns` fallback in TableHeader

**File:** `src/components/job-table.tsx:32`

The separator line width uses `process.stdout.columns` with a fallback of 120. In test environments, `process.stdout.columns` may be undefined or very narrow. The current ink-testing-library tests appear to handle this correctly (the test output shows a ~120-char separator), but this could cause visual issues in very narrow terminals.
