# M04 Acceptance Criteria

> Visual baselines live under `.workflow/m04/screenshots/baseline/`.
> Tests MUST pass `WatchPaths` pointing at `FileManager.default.temporaryDirectory.appendingPathComponent(UUID().uuidString)`. The default test run MUST NOT touch `~/.agent-jobs/`, `~/.claude/scheduled_tasks.json`, or `~/.claude/projects/` (verified by AC-Q-04 static-grep).
> Timing-sensitive ACs (AC-P-01 latency, AC-P-02 debounce, AC-P-03 visibility-pause, AC-P-04 main-thread block) gated behind `AGENTJOBS_PERF=1` per evolution rule E001 — strict spec assertion is the only one in the test, no relaxed fallback.

## Functional (must all pass)

- [ ] **AC-F-01** — `RefreshScheduler` (or equivalent module — name is architect's call, behavior is binding) exposes a `trigger(_ reason: RefreshTrigger)` method. Triggers from any of `.fileEvent(jobsJson)`, `.fileEvent(scheduledTasks)`, `.fileEvent(claudeProjects)`, `.periodic`, `.manual` are all funneled through one debounced path. Verified by unit test injecting all 5 trigger sources and asserting exactly one downstream `refresh()` call when they fire inside one debounce window.
- [ ] **AC-F-02** — A `DispatchSource.makeFileSystemObjectSource` watcher is installed on `WatchPaths.jobsJson` at app start (or at view-model `startWatchers()` call). When the file's content changes (write through the same fd), `RefreshScheduler.trigger(.fileEvent(.jobsJson))` is invoked. Verified by a unit test that creates a temp file, installs the watcher, writes new bytes, and asserts the trigger callback fires within 100 ms.
- [ ] **AC-F-03** — Same as AC-F-02 but for `WatchPaths.scheduledTasks`. Independent verification (separate test).
- [ ] **AC-F-04** — **Atomic-rename safety.** When the watched file is replaced via temp+rename (the writer does `write tmp; rename(tmp, target)` — the `HiddenStore` pattern), the watcher detects the `.delete` or `.rename` event, re-opens a fresh fd within 200 ms, and is observed to fire on a *subsequent* second write. Verified by a unit test that performs two consecutive temp+rename writes through the watcher and asserts two trigger events, with the watcher remaining live throughout.
- [ ] **AC-F-05** — An `FSEventStreamCreate` watcher is installed on `WatchPaths.claudeProjectsDir` (recursive, latency 0.25 s, `kFSEventStreamEventIdSinceNow`). When any `*.jsonl` file under the directory tree is created or modified, `RefreshScheduler.trigger(.fileEvent(.claudeProjects))` is invoked exactly once per debounce window. Verified by a unit test creating a nested `subdir/session.jsonl` and writing to it.
- [ ] **AC-F-06** — A periodic 10 s tick is installed at `startWatchers()` time and triggers `RefreshScheduler.trigger(.periodic)` every 10 ± 0.5 s while the UI is visible. Verified by a unit test with a controllable clock (or by a 30 s real-time test gated behind `AGENTJOBS_PERF=1`) asserting 3 ± 1 ticks in 30 s.
- [ ] **AC-F-07** — When the dashboard window's occlusion state transitions to "not visible" AND the menu-bar popover is closed, the periodic tick is cancelled within 1 s. When visibility returns, an immediate catch-up `refresh()` fires within 250 ms and the periodic tick re-arms. Verified by a unit test driving a `VisibilityProvider` protocol (production reads `NSApplication`; test passes a fake).
- [ ] **AC-F-08** — During the visibility-paused state, file-event watchers remain armed. A write to a watched file during pause still triggers `refresh()` within the AC-P-01 budget. Verified by unit test.
- [ ] **AC-F-09** — `ServiceRegistryViewModel` exposes `isRefreshing: Bool` (already present in M03 — extended to be set true for ALL refresh paths including file-event-driven and periodic) and `lastRefreshError: String?` (new). Both are `@Observable` and drive the `AutoRefreshIndicator`. Verified by view-model unit test asserting `isRefreshing` flips true → false around any refresh trigger and `lastRefreshError` is non-nil iff `discoverAllDetailed().allFailed == true` for the most recent refresh.
- [ ] **AC-F-10** — `ServiceRegistryViewModel.refresh()` mutates `services` in place (assigns the new sorted array to the same `@Observable` property without an intermediate empty-state). Verified by a snapshot test running 10 consecutive refreshes against a stub registry whose output is identical, asserting `services.count` never transiently equals 0 mid-cycle and asserting service `id` ordering is identical across all 10 frames.
- [ ] **AC-F-11** — `WatchPaths` is a public `Sendable` struct with `jobsJson: URL`, `scheduledTasks: URL`, `claudeProjectsDir: URL`. The view-model constructor accepts an optional `WatchPaths`; `nil` resolves to the production paths under `NSHomeDirectory()`. Verified by inspecting the public API + a unit test confirming `init(watchPaths: nil)` resolves the three default paths to the user-home equivalents.
- [ ] **AC-F-12** — `ServiceRegistryViewModel.stop()` cancels all three file watchers, the FSEventStream, the periodic tick, and the visibility observer with no leaked resources. Verified by a unit test that constructs a view model, calls `startWatchers()`, then `stop()`, then asserts the underlying `DispatchSource` references are nil and the periodic `Task` is `isCancelled == true`.
- [ ] **AC-F-13** — Watcher-install failure (e.g., the watched file does not exist) does NOT crash and does NOT prevent other watchers from installing. The failure surfaces as `viewModel.lastRefreshError` being non-nil with a message naming the failed path. Verified by unit test passing a `WatchPaths` whose `jobsJson` points at a non-existent file; the `scheduledTasks` and `claudeProjectsDir` watchers must still install successfully.
- [ ] **AC-F-14** — When a refresh is triggered while one is already in flight (e.g., file event arrives during `await registry.discoverAll()`), the second trigger is held by the debouncer and fires once the first completes; we do NOT execute two concurrent `discoverAll()` calls. Verified by a unit test using a stub registry that blocks on a continuation, firing two triggers, and asserting `discoverAll()` is called exactly twice (not 1, not 3+) over the full sequence.
- [ ] **AC-F-15** — The M03 `optimisticallyStopped` overlay continues to function correctly under the M04 cadence: a stop action's optimistic `.idle` flip is preserved across a refresh that fires within 2 × periodic-interval (20 s under M04). Verified by view-model unit test extending the existing M03 optimistic-overlay test with a 10 s periodic tick interleaved.

## Performance

- [ ] **AC-P-01** — **End-to-end refresh latency.** From the moment a watched file's content is committed to disk (post-`fsync` on the write) to the moment `services` is reassigned on the view model: median ≤ 500 ms over 20 runs on the dev box, p95 ≤ 800 ms. Measured via a unit test that timestamps both events. Strict 500 ms median assertion gated behind `AGENTJOBS_PERF=1`.
- [ ] **AC-P-02** — **Debounce correctness.** When 5 trigger calls arrive within 100 ms, exactly 1 downstream `refresh()` runs, and it runs within 250 + 50 ms of the LAST trigger. Verified by unit test with a stub clock; gated behind `AGENTJOBS_PERF=1`.
- [ ] **AC-P-03** — **Battery / visibility-pause.** Over a 60 s window with the visibility predicate held false (UI hidden), zero periodic `refresh()` calls fire. File watchers may still fire (and do not count against this AC). Verified by unit test counting `discoverAll()` invocations on a stub registry; gated behind `AGENTJOBS_PERF=1`.
- [ ] **AC-P-04** — **Main-thread non-block.** During a refresh triggered while the dashboard is on-screen, no main-thread tick is blocked for more than 16 ms (one frame at 60 Hz). Verified by a unit test that posts a `Date()` checkpoint to the main run loop every 8 ms during a refresh and asserts no inter-checkpoint gap exceeds 16 ms. Gated behind `AGENTJOBS_PERF=1`.
- [ ] **AC-P-05** — **No M02/M03 perf regression.** AC-P-02 from M02 (3 s discovery on reference HW) and AC-P-01/02/03 from M03 (50 ms HiddenStore round-trip; 3 s `refreshNow`; 10 ms filter on 1k services) all remain green when re-run. Tester runs the existing perf suite gated behind `AGENTJOBS_PERF=1` and reports a diff.

## Quality gates

- [ ] **AC-Q-01** — `swift build` is green on the package root.
- [ ] **AC-Q-02** — `swift test` is green; coverage on changed lines (the new `Refresh/` module + view-model wiring + indicator state branches) ≥ 80 %.
- [ ] **AC-Q-03** — Test count net-positive: at least +20 tests over the M03 final count (226). Spans unit (debounce, atomic-rename, visibility-pause, watcher-install, in-place mutation), visual (5 ACs above), and perf (4 ACs above, gated).
- [ ] **AC-Q-04** — **No real `~/` writes from tests.** A `StaticGrepRogueRefs`-style test asserts no test file string-references `.agent-jobs/`, `.claude/scheduled_tasks.json`, `.claude/projects`, or invokes `NSHomeDirectory()` outside of a documented allow-list. Tester verifies the default `swift test` run produces zero file-system writes under `$HOME` (audited via `fs_usage` on a manual run, OR by injecting a `FileManager` test double — architect picks).
- [ ] **AC-Q-05** — No new third-party Swift package dependency added to `Package.swift`. `DispatchSource`, `CoreServices.FSEventStream`, and any debounce primitive used must be Foundation/Combine SDK only.
- [ ] **AC-Q-06** — `ServiceRegistryViewModel.stop()` is called by every test that constructs a view model (or via a test fixture / harness `deinit`), so no test leaks a background `Task` or watcher fd. Verified by a static-grep test asserting every `ServiceRegistryViewModel(` construction in the test target appears in the same file as a `.stop()` call OR uses a `withViewModel` helper.
